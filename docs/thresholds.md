# Bloomwatch — Threshold catalog

Every Good/Fair/Bad judgement threshold used anywhere in the app, in one place, per story 802's acceptance criteria. This is the permanent reference for calibration passes — story 802 is deliberately not a one-time event (principle 3: thresholds are periodically recalibrated by the maintainers as more real logs are seen), so this doc stays current rather than being retired once 802 first ships.

Each threshold's rationale lives as a code comment next to its constant (principle 3) — this table is an index into those comments, not a replacement for them. If a value here and its code comment ever disagree, the code is authoritative; fix this table.

The in-app **Judgement Rationale** page (`#/judgements`) is the user-facing
companion to this file — it explains the same thresholds in plain language,
importing the real exported constants directly so it can't drift out of
sync the way a second hand-copied number could. This file remains the
permanent dev-facing index into each threshold's code-comment rationale;
that page is for end users.

- **GCD utilization: no change.** 81% good / 11% fair / 7% bad across the sample, median 98.3% — strong, real validation that the current 85%/70% bands already fit known-good deep-resto play, the same character of finding 902 made for refresh cadence.

**Calibration review (story 902, 2026-07):** reviewed against ~22 real logs from 16 distinct players, including several from the 2021-2022 TBC Classic launch's Black Temple/Hyjal/Sunwell progression content (talent-confirmed deep-resto, sourced per story 901).

- **Refresh cadence median: no change.** Seven-plus independent elite players consistently land 70-96% of their refreshes in the current 5.5-7s "ideal" bucket (e.g. Kudryavka 96%, Stuuri 95%, Was 94%, Apalistar 94%), and their pooled medians cluster tightly around the 6-7s good band. This is real validation of the existing threshold, not a guess.
- **Refresh cadence histogram buckets unified with the median bands (2026-07-19).** The histogram previously used a separate, coarser 3-way split (<5.5s early / 5.5-7s ideal / >7s late) than the median Good/Fair/Bad judgement it sits next to on the same card, so the chart's own cutoffs and colors didn't match the metric's headline judgement. The histogram now uses the same 4 bands as the median (<5s / 5-6s / 6-7s / >7s). This splits the old single 5.5-7s "ideal" bucket — validated above at 70-96% for elite players — into two separate fair (5-6s) / good (6-7s) sub-bands; that finer split hasn't itself been separately re-validated against the exemplar corpus, only the combined 5.5-7s range and the median. A future calibration pass should confirm the two sub-bands hold up individually rather than assuming the old aggregate finding carries over unchanged.
- **LB3 uptime per target: threshold values unchanged, but the metric's framing was found to be misleading for raid healers.** The 90%/75% bands ARE achievable by real elite play (Elmaskecura hit 91% on one target), so the numbers themselves aren't wrong. But a player juggling more simultaneous targets structurally shows lower _per-target_ uptime than one focused on fewer — evidenced directly by Olklo's own logs (3 tight targets at 64-65% in a BT/Hyjal log vs. 10 scattered targets at 19-71% in a harder Sunwell log, same player). The card now carries a caveat (`LB3UptimeCard`) explaining the metric is strongest for a dedicated tank-healer assignment and shouldn't be weighted heavily by a primarily raid-healing druid. A deeper fix — weighting or highlighting a player's best-maintained target rather than judging all targets equally — is real but overlaps with story 903's per-fight role/assignment awareness and story 904's rollup-policy work; deliberately not attempted here to avoid scope creep into stories not yet done.

**Revised by direct request, 2026-07-20 (not a new corpus calibration):**

- **LB3 uptime per target: good/fair boundary loosened 90%/75% → 80%/60%.** Requested directly.
- **Per-target LB3 judgements are now reduced to one representative judgement via `weightedMedianJudgement`** (weighted by each target's own tracked-uptime window) **before joining Lifebloom Discipline's other siblings**, instead of every target being folded in flatly alongside refresh cadence / blooms / re-stack tax. Motivating real example: `mtRh3kJ9YMLazyvQ` fight 44 (druid Olklo) — 3 targets at 96%/75%/94% uptime (good/fair/good under the new bands), no target genuinely bad, used to read "fair" under flat worst-of; now reads "good" since the one fair target carries only ~1/3 of the total tracked weight. See the compounding-factors section below for the general rule.
- **Concurrent LB3 targets gained a real judgement** (story 205, resolving one of story 914's four bullets): good when 2+ targets held LB3's 3rd stack for at least 50% of the fight, otherwise unjudged — never fair or bad, since anything below that bar may simply reflect a raid-healing assignment rather than weaker play. Folds into the Lifebloom Discipline epic verdict as a 5th sibling (a `null` judgement is filtered out, so it can only ever help).

**Pre-pull carry-in lookback resolution (story 915, 2026-07-20):**

- **Carry-in targets now resolved via bounded lookback fetch, rather than defaulting to confident "bad" when ambiguous.** When a target's Lifebloom timeline opens mid-stream (first event ≠ `"open"`, indicating pre-pull stacking or a cross-wipe WCL fight-ID boundary), a secondary 60-second pre-`fightStart` Buffs-events fetch (omitting the `fightIDs` filter to capture carry-in state from before the current WCL fight boundary — confirmed live against report `DRtXV4ChA2Kw3c81` fights 24→25, where a `fightIDs`-scoped query returns zero events) is performed conditionally by each consumer (hooks, `calibrateReport.ts`). The merged timeline is re-analyzed to resolve the target's true carried-in stack state. If a genuine `applybuffstack` + `refreshbuff` sequence emerges in the lookback window, the target is judged normally from that resolved point; if ambiguity persists after the lookback (motivating real example: `DRtXV4ChA2Kw3c81` fight 84, druid Stuuri, targets Jeloviina/id 30 and Hulina/id 6, each showing 100% any-stack Lifebloom uptime over the full 240s fight with zero `applybuff` events in any available window), the target is excluded from judgement entirely — `lb3UptimePct` and concurrent-targets metrics both treat it as unjudgeable rather than falsely confident "0%, bad." This eliminates a real false-negative pattern where an expertly-maintained pre-stacked target was misread as grossly neglected, while preserving the default conservatism when no lookback data can resolve the ambiguity.

**Faerie Fire duty drag check (story 917, 2026-07-23), phase 1 only — no mitigation designed or implemented yet.** Compared FF-duty vs. non-FF-duty fights across the local calibration corpus (44 reports, every Balance-leaning-bucket report:druid pair meeting a real-healing-participation filter — see `docs/backlog.md`'s story 917 entry for the full corpus description and its caveat that only `likely-dreamstate-full` druids are represented so far) via `npm run analyze:ff-drag`. **Methodology caveat:** the comparison pools every qualifying druid's fights into two global groups and compares aggregate medians, not a within-druid, within-bucket comparison as the design spec intended — so a baseline skill or roster difference between the two groups (dominated by different druids) can't be ruled out as a confound; a future within-druid comparison would be more rigorous.

- **Re-stack tax: real, substantial drag.** FF-duty fights median 9 casts (n=228) vs. non-FF-duty median 3 (n=45) — roughly triples, plausibly crossing `judgeRestackTax`'s duration-scaled good/fair boundary into fair/bad on a typical multi-minute fight. Flagged as this story's clearest mitigation candidate for a future phase 2 (not yet designed or implemented). **Caveat:** this is a comparison of raw cast counts, not `judgeRestackTax`'s duration-normalized judgement, so part of the gap could reflect FF-duty fights simply running longer on average; phase 2 should re-check against the judged bands before committing to a mitigation magnitude.
- **Accidental blooms: a real but modest effect.** FF-duty median 2 (n=228) vs. non-FF-duty median 1 (n=45), both landing in the same fair band (good 0 / fair 1–2 / bad ≥3) at opposite ends of it.
- **Refresh cadence: a small difference, inconclusive.** FF-duty median 5428.5ms (n=180) vs. non-FF-duty median 5027ms (n=29), both in the same fair band (6–7s good / 5–6s fair / <5s or >7s bad); the non-FF-duty median sits at the edge of the bad boundary while the FF-duty median sits comfortably mid-fair, so this doesn't clearly point toward FF duty being the worse case.

Story 917 stays 🔲 Todo — phase 2 (mitigation design, informed by these findings) is a separate future spec/plan.

**Calibration review (story 909, 2026-07):** reviewed against story 901's 20-report deep-resto exemplar corpus (`docs/testing.md`'s "Known real 2021-2022 TBC Classic reports" table), regenerated fresh via `scripts/calibrate.ts` after story 013's non-raid-zone contamination fix and the Lifebloom any-stack-uptime carry-in fix. 134 real kill-fights (duration > 30s) from the 20 talent-confirmed-`isRestoSpec` druids.

- **Rejuvenation clip share: no change.** 93% good / 5% fair / 1% bad across the sample, median 0% clips, p90 only 3.3% — strong real validation that the current <5%/5-15% bands already fit known-good deep-resto play.
- **Swiftmend wasteful share: good boundary widened from an exact-zero good band to <40%, bad boundary widened 25% → 80%.** The old thresholds convicted the median real player: pooled across 440 real Swiftmend casts, elite players land 58.2% wasteful overall (median 63.6% per fight), only 13% of fights hit the old zero-tolerance good bar. Digging into why: the "wasteful"-classified casts have a median 10.6s of HoT still remaining and a median target HP of 73% (min 51%) — real skilled play routinely bursts Swiftmend onto a healthy-ish target with a live HoT rather than waiting for either the HoT to nearly expire or the target to cross the 50% "emergency" line, which this metric's classification can't tell apart from careless spam. The new <40%/40-80%/>80% band (32%/39-50%/18-28% real split depending on minimum-sample filtering) reflects that reality without abandoning judgement entirely. **Flagged, not fixed:** `classify()`'s own emergency-HP cutoff (currently 50%) is a plausible root cause worth a future look — real "wasteful" casts cluster well above it — but changing the classification logic itself is out of this story's scope (a threshold-value recalibration, per its acceptance criteria), so only the wasteful-share band moved here.
- **Downranking flagged-count: no change.** 88% good (0 flags) / 12% fair (≥1 flag) across the sample; max flags seen in any real fight was 1, consistent with the existing "bad is structurally unreachable" note. The underlying >50% direct-overheal flag cutoff (a separate threshold, not itself in this story's scope) sits almost exactly at the real p75 of max-rank overheal-by-spell-group (median 36%, p75 51%, p90 62%) — a defensible position, additional support for leaving both thresholds as-is.

**Revised by direct request, 2026-07-20 (not a new corpus calibration):**

- **Swiftmend gained a second judgement, utilization** (usage vs. its 15s-cooldown availability, previously informational context only): good ≥75%, fair 50-75%, bad <50%. Folds into Spell Discipline's epic verdict alongside its existing wasteful-share judgement, gated the same way (druid's build must reach Swiftmend's 30-Restoration talent requirement). **Corrected same-day, post-final-review:** the initial 75%/50% bands (copied from Nature's Swiftness) proved unreachable in practice — story 909's real corpus averages only ~3.3 Swiftmend casts per fight, far below what 75% demands on a typical fight's ~20 windows — softened to 50%/25%, still provisional pending real calibration data specific to this metric.
- **Nature's Swiftness gained a real judgement** (story 304, resolving one of story 914's four bullets), same bands as Swiftmend above, with one exception: a fight with only 1 available window (under 3 minutes) can only ever land on 0% or 100% utilization, and holding Nature's Swiftness in reserve for a real emergency that may just not occur is reasonable on a short fight — so 0 casts there reads fair, not bad. Folds into Spell Discipline's epic verdict, gated by the druid's build reaching Nature's Swiftness's 20-Restoration talent requirement.

**Regrowth clip share gained a real judgement for every archetype except deep-resto (story 914, 2026-07-21), pooled against the local 100-report calibration corpus (`calibration-data/`, kill-fights >30s), tagged via `docs/calibration-archetypes.json`:**

- deep-resto's exemption is scoped to why it exists — Tree of Life (Restoration ≥41) forces Regrowth into being a druid's only non-cooldown direct heal, which is a structural fact of every non-deep-resto bucket too (`classifyBucket` in `src/report/archetypeDetection.ts` returns deep-resto exactly when Restoration ≥41, so every other bucket is Restoration <41 by construction) — the exemption's premise simply doesn't hold once Healing Touch is available with no form-swap tax.
- Pooling every non-deep-resto tagged bucket together (n=163 fight-rows, 124 of them likely-dreamstate-full — the dominant real archetype in this corpus) against Rejuvenation's existing <5%/5-15%/>15% bands landed a clean, non-degenerate split: 69% good / 9% fair / 22% bad, median 0%, p75 11.1%, p90 40% — good is genuinely modal and bad a real minority, the same shape story 909 validated for Rejuvenation's own clip share, so Rejuvenation's exact bands were reused rather than deriving new ones.
- For contrast, deep-resto's own Regrowth clip share pooled to 56% good / 20% fair / 24% bad (median 2.7%, p75 14.3%, p90 36%) under the same bands — deep-resto stays informational regardless, since the exemption is about _why_ clipping happens there, not about whether the numbers happen to look similar.
- `unknown-no-talent-data` (talent read failed) is deliberately excluded from the new judgement and grouped with deep-resto instead — a failed read can't rule out the exemption applying (matching story 903d's identical reasoning for why that bucket is left unflagged by the onboarding notice).
- Wired through the same 3 call sites `overhealTable.ts`'s per-archetype split already established: `HotClipDetectionCard`, `useSpellDisciplineSummary` (via `summarizeSpellDiscipline`'s `mixedJudgement`), and `scripts/lib/calibrateReport.ts`.

**HoT-tick overheal confirmed informational, no threshold added (story 914, 2026-07-21).** Pooled Rejuvenation and Regrowth's HoT-tick overheal against the full local corpus (`calibration-data/`, grown to 120 reports for this review, tagged via `docs/calibration-archetypes.json`): deep-resto median 54%/57% (Rejuvenation/Regrowth-HoT), dreamstate median 61%/64%, with a real p10-p90 spread of 35-45+ points per archetype bucket — unlike Bloom overheal's tight 72-74% clustering (story 905) that justified giving it a real threshold, this spread never tightened as the corpus grew. The decisive check: for druids with 3+ fights, average **within-player** fight-to-fight spread (41 points Rejuvenation, 50 points Regrowth-HoT) exceeds average **between-player** spread (27/31 points) — the same druid varies more across their own fights than typical players differ from each other. This held up after adding 20 reports from a second guild's healer (self-labeled elite, consistently deep-resto talented at 0/0/61, high WCL parses) specifically to stress-test it: that player's own Rejuvenation overheal ranged 19-84% within one 13-fight report alone. That's the signature of an encounter-driven number (movement, add phases, how much raid-wide healing overlaps a pull) rather than a stable per-player skill trait — the original "inherent to HoTs during raid healing, don't judge" call is confirmed as-is; no code change.

**Calibration review (story 905, 2026-07):** mana economy's whole-report rollup was found to be driven almost entirely by overheal (204/393 real fight-rows bad; mana curve, consumables, and Innervate were all reasonably distributed on their own). Reviewed against the same story-901 exemplar corpus 902/908 used for deep-resto data, plus the broader talent-tagged `calibration-data/` corpus for dreamstate (no dedicated dreamstate exemplar hunt exists yet — see below).

- **Bloom (Lifebloom) overheal: good boundary revised 40% → 80%, bad boundary revised 70% → 90%, unchanged across archetypes.** Real deep-resto and dreamstate exemplars both cluster at 72-74% median overheal — the old 70% bad line convicted the median real player of any archetype. The new bands land roughly 60% good / 25% fair / 15% bad across the pooled sample.
- **Regrowth-direct overheal split by archetype for the first time.** deep-resto's real median (31%) sat close to the old boundary already, so its band moved modestly (good < 30% → < 38%, bad > 50% → > 60%). Dreamstate's real median (50%, p75 84%) is structurally much higher, so its band is substantially wider (good < 60%, bad > 85%) — **this number is provisional**, calibrated against the broader talent-tagged corpus rather than a behaviorally-validated exemplar set (no dreamstate equivalent of story 901's deep-resto exemplar hunt exists yet); a future "901 but for dreamstate" story would strengthen it. Every other archetype bucket (mostly-resto, mostly-balance, restokin-shaped, other-unclassified, unknown-no-talent-data) uses the deep-resto band as a fallback default, since those builds aren't well-supported by this tool yet (story 903d) and this review doesn't manufacture a new precision claim about them.
- **Healing Touch and Swiftmend overheal: no change.** Both already fit the existing 30%/50% bands well in every archetype bucket (median 0-4% overheal) — real validation, not a guess.

**Calibration review (story 911, 2026-07):** story 905's own "mana curve, consumables, and Innervate were all reasonably distributed on their own" was a passing observation, not a real percentile finding — this pass gives those three metrics the same rigor 905 gave overheal, reviewed against the story-901 20-report deep-resto exemplar corpus (regenerated fresh via `scripts/calibrate.ts` first), corroborated against the broader 100-report talent-tagged corpus.

- **Ending mana %: no change.** 143 judged kill-fights ≥90s: good 40% / fair 38% / bad 22%, median 34.5%, p75 62.5% — the good band contains the real median almost exactly and p75 sits just under the bad boundary, a solid fit on both corpora (broader corpus within 3 points on every figure).
- **Consumable throughput — Mana Potion: no change.** 210 mana-constrained fights: good 59% / fair 28% / bad 13%, used-vs-floor delta median 0 — real elite players meet the floor almost exactly on the median fight.
- **Consumable throughput — Rune: floor interval widened 120s → 300s (`RUNE_FLOOR_INTERVAL_MS`, now split from Mana Potion's own `POTION_FLOOR_INTERVAL_MS`).** Real elite play used a Rune in only 10% of mana-constrained exemplar fights regardless of length (mean use 0.03-0.32 even in 4-8min fights, never scaling with duration like Mana Potion does) — the old shared 120s floor scored real elite players good only 20% of the time (32% bad), the median real player convicted outright, the same pattern 909 found broken in Swiftmend's old wasteful-share band. This tracks real TBC context: Dark Rune needs Enchanting-crafted reagents and Demonic Rune is a scarce vendor item, both cost health to use, unlike cheap always-stackable Mana Potions. The new 300s interval fits real data much better: good 80% / fair 20% / bad 1% (exemplar), good 86% / fair 14% / bad 0% (broader corpus).
- **Innervate audit: no change**, with one flagged-not-fixed caveat. 160 judged-relevant fights: good 75% / fair 6% / bad 19%. The ally-cast "wasted on non-mana target" bad path never fired in real data (0/28) and the self-cast late-timing split (90% on-time / 10% late) is a sensible non-degenerate spread — both structurally sound, no change. The entire bad bucket comes from "no Innervate cast in a mana-constrained fight ≥3min," and 77% of those cases cluster within 3.0-4.4 minutes — plausibly Innervate still on cooldown from the previous pull (its real TBC cooldown is 6 minutes) rather than a genuine miss. `computeInnervateAudit` has no cross-fight cooldown tracking, unlike `deathForensics.ts`'s `isReady`; fixing that properly needs real cross-fight state, not a threshold-value move, so it's flagged for a future story rather than fixed here — raising the 3-minute cutoff alone wouldn't resolve the ambiguity anyway (a longer fight can still have Innervate on cooldown the entire time).

**Faerie Fire duty drag check (story 917, 2026-07-23), phase 1 only — no mitigation designed or implemented yet.** Same corpus and method as the Lifebloom discipline section above (44 reports, `npm run analyze:ff-drag`, `likely-dreamstate-full` druids only so far) — including the same pooled-not-paired methodology caveat noted there (comparison is across two global druid groups, not each druid against their own baseline).

- **Ending mana %: checked, no mitigation needed.** FF-duty median 41.587...% (n=228) vs. non-FF-duty median 47.465...% (n=43) — both land in `judgeManaBand`'s fair 40–70% band, but FF-duty's is lower, closer to the 5–40% good boundary that rewards spending mana rather than hoarding it (the only bad zone is >70%, hoarding). Mana spent on Faerie Fire doesn't push this metric toward a worse judgement.
- **Consumable throughput: checked, no mitigation needed.** Used-minus-floor delta median is 0 for both FF-duty (n=228) and non-FF-duty (n=45) fights — no measurable difference. Caveat: many fights in this corpus have `expectedFloor: 0` (short fights requiring no consumables at all), so this is a real "no change" finding rather than a definitive rule-out of any mana-pressure effect specific to this metric.

## Compounding factors (not thresholds themselves, but shape how harsh the tool reads)

Several structural choices amplify the effect of any single strict threshold, independent of whether that threshold's own value is right:

- **Worst-of aggregation, two levels deep — revised 2026-07-19 (story 904), epic level revised again 2026-07-19.** `lb3Uptime.ts` doesn't combine per-target judgements at all — each target's LB3 uptime gets its own independent `judgeThreshold` call, with no folding across targets in that file. The actual per-target-to-epic combination happens one layer up, in `epicSummary.ts`'s `summarizeLifebloomDiscipline`, which — like 4 of its siblings there (`summarizeGcdEconomy`, `summarizeSpellDiscipline`, `summarizeManaEconomy`) plus `prepHygiene.ts`'s `computePrepHygiene` — no longer uses strict worst-of: all 5 use `mixedJudgement` (`src/metrics/judgement.ts`), which reads a good+bad sibling mix as `fair` rather than a flat `bad`, falling back to worst-of otherwise — see the next bullet. `702`'s whole-report dashboard doesn't compound either of these a third time across a whole raid night: it takes a duration-weighted median of each epic's per-fight judgements (`weightedMedianJudgement` in `src/metrics/judgement.ts`, shared with `scripts/lib/rollup.ts`'s CLI rollup), with a fight-count breakdown shown alongside each chip so a user can still see what drove the result. Motivating corpus data: GCD economy was 33% good / 27% fair / 39% bad per fight but 0% good / 9% fair / 91% bad under the old worst-of rollup; spell discipline was 70% good per-fight but only 35% good at rollup.
- **Calibration corpus is narrow.** `docs/testing.md`'s known-report table is drawn almost entirely from one guild's raid logs (used originally to validate event-parsing assumptions, not to represent a range of skill levels), plus one unrelated report added for spec-detection coverage. None were sourced as _known-good, well-regarded_ play. Story 802's calibration pass needs additional reports from strong resto druids specifically for this purpose — see the "known-good" gap called out in the calibration-pass task list.
- **Whole-report rollup: a good-and-bad mix now reads as fair, not a flat bad — added 2026-07-19.** Requested directly: `weightedMedianJudgement` (`src/metrics/judgement.ts`) still buried a mixed result under the plain duration-weighted median above — e.g. 1 good fight, 2 fair, and 8 bad fights of similar length used to read as a flat "bad" for the whole epic, even though at least one fight genuinely went well. Whenever an epic's ready fights include at least one good judgement and at least one bad judgement, the rollup now reports "fair" outright, ahead of the median calculation, regardless of which side dominates by duration. This is deliberately symmetric — a mostly-good epic with a single bad outlier fight is also capped at "fair" rather than "good" — since the rule cares about the mere presence of both extremes, not which one wins the weighting.
- **Epic verdicts also read fair (not a flat bad) when a sibling-metric mix has both good and bad — added 2026-07-19.** Requested directly: Spell Discipline with 2 good metrics (HoT clip detection, downranking discipline) and 1 bad (Swiftmend quality audit) used to read a flat "bad" for that fight, the same over-harsh pattern the whole-report rollup bullet above already fixed one level up. `mixedJudgement` (`src/metrics/judgement.ts`) now governs 6 multi-metric functions across three files — `epicSummary.ts`'s `summarizeGcdEconomy`, `summarizeLifebloomDiscipline`, `summarizeSpellDiscipline`, and `summarizeManaEconomy`, `prepHygiene.ts`'s `computePrepHygiene`, and `consumableThroughput.ts`'s `computeConsumableThroughput`: whenever the sibling metrics being combined include at least one good and at least one bad, the result is "fair" outright; otherwise it falls back to strict worst-of unchanged (a fair-only mix with no bad present, e.g. 2 good + 1 fair, still reads "fair" exactly as before — a full majority-by-count median was considered and rejected because it would have also turned that case "good").
- **Consumable throughput's own row-combination fixed to match every other multi-part judgement — added 2026-07-20.** `computeConsumableThroughput` (`src/metrics/consumableThroughput.ts`) combined its Mana Potion and Rune rows via strict `worstJudgement`, the one remaining multi-part metric in the codebase not using `mixedJudgement`'s good+bad-reads-fair rule. Now consistent: a good potions row and a bad runes row (or vice versa) reads fair, not a flat bad.
- **Swiftmend quality audit's own header chip now combines both of its judgements, with efficiency weighing heavier — added 2026-07-20.** Requested directly: `SwiftmendAuditCard`'s header chip used to show only the wasteful-share ("efficiency") judgement, with the separate utilization judgement shown only as its own small inline chip next to the usage sentence. `combineSwiftmendCardJudgement` (`src/metrics/swiftmendAudit.ts`) now combines them for the header: efficiency is the anchor, and a bad utilization can only drag it down one notch (good → fair, fair → bad) — a good or fair utilization leaves efficiency's own judgement unchanged, and utilization never pulls efficiency up. This is a different, card-scoped combination from `epicSummary.ts`'s `summarizeSpellDiscipline`, which already independently folds both Swiftmend judgements (plus HoT clip detection and downranking discipline) into the Spell Discipline epic verdict via `mixedJudgement` — the two combinations serve different purposes (this card's own header chip vs. the whole epic's verdict) and aren't meant to produce the same result.
